Down the gVisor Rabbit Hole: A Sunday Afternoon Deep Dive

2026-06-28
#kubernetes  #cri-o  #gvisor  #linux  #security 

It’s a quiet Sunday afternoon. The kind of afternoon where you have a hot cup of cha on the desk, a terminal window open, and the headspace to write a new post—especially now that I’m starting to make some real headway on a new CRI-O feature. This journey actually began when I decided to get back into open-source software (OSS) following a break after the birth of my daughter.

I reached out to Peter Hunt (@haircommander), one of the lead maintainers of CRI-O, to see if there were any interesting challenges or areas that needed some love. He pointed me toward a long-standing integration puzzle: getting gVisor to play nicely with CRI-O, specifically around how the runtime handles multi-container pods.

That was all the spark I needed. I accepted the challenge and started digging into the gVisor documentation. Here is a breakdown of how gVisor works, the specific lifecycle issue we need to solve, and the fascinating design patterns behind it.

What actually is gVisor?#

To understand the problem, we have to look at the classic container security dilemma.

Normally, containers aren’t virtual machines. They share the host operating system’s kernel. Standard runtimes like runc use Linux namespaces and cgroups to isolate processes, but the system call interface is still shared. If a containerized application finds a zero-day vulnerability in the Linux kernel, it can make a malicious system call and potentially break out of the container to compromise the host.

On the other end of the spectrum, you have Virtual Machines (VMs) or microVMs (like Kata Containers or Firecracker). They run a full guest OS with their own kernel. This is highly secure, but it comes with a tax: slower startup times, higher memory footprint, and less efficient resource sharing.

gVisor sits right in the sweet spot.

Instead of virtualizing the hardware, gVisor virtualizes the kernel. It is a user-space kernel written in Go that intercepts all system calls made by the containerized application. To the application, gVisor looks and acts exactly like a normal Linux kernel. But under the hood, gVisor handles those system calls in user space without letting them touch the host kernel directly.

If the application makes a syscall, gVisor catches it, processes it in a non-privileged environment, and only makes safe, filtered calls to the host kernel when absolutely necessary.

The Architecture: Sentry, Gofer, and runsc#

gVisor splits its responsibilities across three core components:

    end
    Sentry -->|9P Protocol| Gofer[Gofer <br/> File System Mediator]

    Sentry -->|Filtered Syscalls| HostKernel[Host Kernel]

    runsc -->|Manages| Sentry
    runsc -->|Manages| Gofer

1. The Sentry#

The Sentry is the heart of gVisor. It is a complete, multi-threaded user-space kernel written in Go. It implements the vast majority of the Linux system call table (over 300 syscalls).

  • When the application wants to allocate memory, create a socket, or start a thread, it makes a system call.
  • The Sentry intercepts this call using a virtualization platform (like KVM or ptrace).
  • It then executes the kernel logic for that syscall entirely inside the Sentry process in user space. The host kernel has no idea the application even tried to make that call.

2. The Gofer#

File system access is historically one of the most common vectors for container escapes. To keep the Sentry as isolated as possible, it is not allowed to open host files directly.

Instead, gVisor uses a helper process called the Gofer. The Sentry talks to the Gofer over a Unix domain socket using a custom version of the 9P wire protocol (often called Omni9P). When the application wants to read a file, the Sentry asks the Gofer to open and read it, and the Gofer sends the data back. Even if an attacker somehow compromises the Sentry, they are still trapped inside a process that cannot directly access the host filesystem.

3. runsc#

To make all of this work seamlessly with the existing container ecosystem, gVisor provides runsc (Run Sandboxed Container). runsc is an OCI-compliant runtime CLI. It implements the same command-line interface as runc (runsc start, runsc state, runsc delete, etc.).

Because it adheres to the OCI spec, container orchestrators like Kubernetes—via container runtimes like CRI-O or containerd—can swap runc for runsc without needing to change how they manage containers.

The Specific Challenge: Issue #10313#

So, what is the actual issue we need to solve?

It turns out that while gVisor works reasonably well with containerd, it trips up under CRI-O when dealing with Kubernetes initContainers (tracked in gVisor Issue #10313). Credit for much of the initial discovery and troubleshooting goes to the original author of the issue, @hbhasker, and the contributors in the thread who did the heavy lifting of mapping out the problem.

Even so, actually solving this is still a massive challenge for me. This is my very first time reading through the gVisor codebase, and navigating a user-space kernel written in Go is a whole different beast.

In gVisor’s multi-container design, the runtime expects a single “root” container to represent the sandbox. The Sentry runs inside this root container and manages the lifecycle of all other sub-containers (which are run as stub processes inside the same Sentry instance).

Here is where the assumption breaks:

  1. When a Kubernetes Pod starts, the Container Runtime Interface (CRI) creates the sandbox.
  2. If the Pod has initContainers, CRI-O creates and starts the first initContainer before any application containers are created.
  3. Because the initContainer is the first container created in the sandbox (after the pause container), gVisor treats it as the root container.
  4. The initContainer runs its task, completes successfully, and exits.
  5. Because the “root” container has exited, gVisor’s lifecycle management gets confused. When CRI-O subsequently tries to start the actual application containers, the sandbox is in a broken state because the Sentry’s primary container process is gone.

Compounding the issue, the CRI spec doesn’t explicitly flag to the OCI runtime whether a container is an initContainer or a standard application container. To CRI-O and runsc, it’s just another container creation request.

Figuring out how to decouple the Sentry’s lifecycle from the first application-level container—or teaching runsc how to handle containers that run to completion and exit without tearing down the sandbox—is the core puzzle I’ll be digging into.

Where is this actually used?#

As I was reading about gVisor, I started wondering: who is actually running this in production? It turns out, some of the most widely used cloud services rely on gVisor for security:

  • Google Cloud Run: When you deploy a serverless container on Cloud Run, Google is running your untrusted code on shared infrastructure. Under the hood, Cloud Run uses gVisor to ensure that customer containers are completely sandboxed from one another and the underlying host.
  • GKE Workload Isolation (GKE Sandbox): Google Kubernetes Engine offers a “Sandbox” mode. If you’re running multi-tenant workloads or untrusted third-party applications, GKE wraps them in gVisor.
  • LLM & AI Code Execution: This is a massive, modern use case. When you build AI agents or LLM platforms (like ChatGPT’s Advanced Data Analysis) where the LLM writes Python code and executes it on the fly, you must run that code in an extremely secure sandbox. gVisor is the perfect fit because it starts instantly (unlike a VM) but prevents the LLM-generated code from doing anything malicious to the host.

The "Aha!" Moment: The Interception Pattern#

While thinking about gVisor’s system call interception model, I realized that this architecture is part of a beautiful, recurring design pattern in computer science. Once you see it, you start seeing it everywhere:

Technology What it Intercepts What it Translates To Purpose
gVisor Container Linux Syscalls User-space Go logic / Safe Host Syscalls Security & Sandboxing
WSL 1 Linux Syscalls Windows NT Kernel APIs Developer Convenience
Rosetta 2 x86_64 Instructions ARM64 Instructions Hardware Transition
Wine / Proton Windows Win32 & DirectX APIs POSIX & Vulkan/OpenGL APIs Compatibility (Gaming on Linux)
  • WSL 1 (Windows Subsystem for Linux 1): Before WSL2 moved to a lightweight Hyper-V VM, WSL1 worked by intercepting Linux system calls and translating them on the fly into Windows NT API calls.
  • macOS Rosetta 2: Apple’s translation layer intercepts x86_64 instructions from legacy Mac apps and translates them into ARM64 instructions that run on Apple Silicon.
  • Wine & Proton: They intercept Windows system and graphics calls (like DirectX) and translate them to Linux system calls and Vulkan/OpenGL APIs, allowing Windows games to run smoothly on Linux.

It’s the same mind-bending concept: instead of running the actual target environment, you build a translator at the boundary line. You intercept the interface, emulate the behavior, and map it to a different backend.
